# Tree Session Manager

Tree-structured conversation sessions for picoclaw, inspired by [pi-mono](https://github.com/badlogic/pi-mono).

## Overview

The tree session manager (`pkg/session/tree/`) replaces the legacy flat message history with a tree-structured JSONL format. Conversations are stored as append-only JSONL files where each entry is linked to its parent via `id`/`parentId` references, forming a tree.

This enables:
- **Branching** -- explore alternate conversation paths without losing history
- **Compaction** -- summarize old context to stay within the LLM context window
- **Labels** -- bookmark interesting points in the conversation
- **Custom entries** -- extensions can store data in the session (with or without LLM context)
- **Session metadata** -- display names, parent sessions, version tracking

## Quick start

```go
import "github.com/sipeed/picoclaw/pkg/session/tree"

// Create a new persisted session
sm := tree.Create("/path/to/project", "/path/to/sessions")

// Append messages
sm.AppendMessage(tree.AgentMessage{Role: "user", Content: "Hello"})
sm.AppendMessage(tree.AgentMessage{Role: "assistant", Content: "Hi!"})

// Build context for LLM
ctx := sm.BuildSessionContext()
// ctx.Messages contains the conversation from root to current leaf

// Branch from an earlier point
entries := sm.GetEntries()
sm.Branch(entries[0].ID) // go back to first entry
sm.AppendMessage(tree.AgentMessage{Role: "user", Content: "Different question"})
// now the tree has two branches from the first entry
```

## SessionStore compatibility

The `tree.Adapter` implements `session.SessionStore`, so it works as a drop-in replacement for the agent loop:

```go
import "github.com/sipeed/picoclaw/pkg/session/tree"

adapter := tree.NewAdapter("/path/to/sessions")

// Standard SessionStore interface
adapter.AddMessage("session-key", "user", "hello")
history := adapter.GetHistory("session-key")

// Access the full tree API when needed
sm := adapter.Manager("session-key")
sm.Branch(entryID)
tree := sm.GetTree()
```

## File layout

### Session directory

```
~/.picoclaw/workspace/sessions/tree/<sanitized_key>/
    2026-03-14T10-00-00-000000000Z_<uuid>.jsonl
    2026-03-14T11-30-00-000000000Z_<uuid>.jsonl
```

Each session key gets its own subdirectory. Inside, each `.jsonl` file is a single session.

### JSONL structure

Each line is a JSON object with a `type` field. The first line is always the session header.

```
{"type":"session","version":3,"id":"<uuid>","timestamp":"...","cwd":"/path"}
{"type":"message","id":"a1b2c3d4","parentId":null,"timestamp":"...","message":{"role":"user","content":"Hello"}}
{"type":"message","id":"b2c3d4e5","parentId":"a1b2c3d4","timestamp":"...","message":{"role":"assistant","content":"Hi!"}}
```

## Entry types

All entries share these common fields:

| Field | Type | Description |
|-------|------|-------------|
| `type` | string | Entry type discriminator |
| `id` | string | Unique 8-char hex ID |
| `parentId` | string\|null | ID of parent entry (null for roots) |
| `timestamp` | string | ISO 8601 timestamp |

### `message`

A conversation message (user, assistant, tool result).

```json
{"type":"message","id":"a1b2c3d4","parentId":"prev1234","timestamp":"...","message":{"role":"user","content":"Hello"}}
```

The `message` field is an `AgentMessage` with these roles:
- `user` -- user input
- `assistant` -- LLM response (may include `provider`, `model`, `tool_calls`)
- `tool` -- tool result (includes `tool_call_id`)
- `system` -- system message

### `thinking_level_change`

Records a change in the thinking/reasoning level.

```json
{"type":"thinking_level_change","id":"...","parentId":"...","timestamp":"...","thinkingLevel":"high"}
```

### `model_change`

Records a switch to a different LLM model.

```json
{"type":"model_change","id":"...","parentId":"...","timestamp":"...","provider":"anthropic","modelId":"claude-sonnet-4-5"}
```

### `compaction`

Marks a context compaction point. Messages before `firstKeptEntryId` are replaced by the summary.

```json
{"type":"compaction","id":"...","parentId":"...","timestamp":"...","summary":"User discussed X, Y, Z...","firstKeptEntryId":"c3d4e5f6","tokensBefore":50000}
```

Optional fields:
- `details` -- extension-specific metadata (JSON)
- `fromHook` -- true if generated by an extension

### `branch_summary`

Records a summary of an abandoned conversation branch. Created when branching with `BranchWithSummary()`.

```json
{"type":"branch_summary","id":"...","parentId":"a1b2c3d4","timestamp":"...","fromId":"a1b2c3d4","summary":"Previous branch discussed approach A..."}
```

### `custom`

Extension-specific data. Does **not** participate in LLM context.

```json
{"type":"custom","id":"...","parentId":"...","timestamp":"...","customType":"my-extension","data":{"count":42}}
```

### `custom_message`

Extension-injected message. **Does** participate in LLM context.

```json
{"type":"custom_message","id":"...","parentId":"...","timestamp":"...","customType":"my-extension","content":"Injected context...","display":true}
```

- `display: true` -- show in UI with distinct styling
- `display: false` -- hidden from UI, but still sent to LLM

### `label`

User-defined bookmark on an entry.

```json
{"type":"label","id":"...","parentId":"...","timestamp":"...","targetId":"a1b2c3d4","label":"checkpoint-1"}
```

Set `label` to `null` or `""` to clear.

### `session_info`

Session metadata (display name).

```json
{"type":"session_info","id":"...","parentId":"...","timestamp":"...","name":"Refactor auth module"}
```

## Tree structure

Entries form a tree via `parentId` references:

```
[user: "Hello"]                parentId=null (root)
    |
    +-- [assistant: "Hi!"]     parentId=root
    |       |
    |       +-- [user: "How?"] parentId=assistant  <-- current leaf
    |
    +-- [branch_summary]       parentId=root
            |
            +-- [user: "Alt"]  parentId=branch_summary  <-- alternate branch
```

Key concepts:
- **Root entries** have `parentId: null`
- **Leaf** is the current position in the tree (the most recent append target)
- **Branch** moves the leaf to an earlier entry; new appends create a new child
- **Path** from leaf to root is the active conversation that gets sent to the LLM

---

## Context building

`BuildSessionContext()` walks from the current leaf to the root and produces the message list for the LLM.

### Without compaction

All messages along the path from root to leaf, in order.

### With compaction

When a `CompactionEntry` is on the path:

1. Emit `CompactionSummaryMessage` (the summary)
2. Emit messages from `firstKeptEntryId` up to (but not including) the compaction entry
3. Emit messages after the compaction entry

This preserves recent context while replacing old messages with a summary.

### Settings extraction

As the path is traversed, `BuildSessionContext` also extracts:
- **ThinkingLevel** -- from the latest `thinking_level_change` entry on the path
- **Model** -- from the latest `model_change` entry or assistant message with provider info

## API reference

### Factory functions

| Function | Description |
|----------|-------------|
| `Create(cwd, sessionDir?)` | New persisted session |
| `Open(path, sessionDir?)` | Open existing `.jsonl` file |
| `ContinueRecent(cwd, sessionDir?)` | Continue most recent or create new |
| `InMemory(cwd?)` | No file persistence |

### Write methods

All write methods return the new entry's ID.

| Method | Description |
|--------|-------------|
| `AppendMessage(msg)` | Append a conversation message |
| `AppendProviderMessage(msg)` | Append a `providers.Message` (auto-converts) |
| `AppendThinkingLevelChange(level)` | Record thinking level change |
| `AppendModelChange(provider, modelId)` | Record model change |
| `AppendCompaction(summary, firstKeptId, tokensBefore, details, fromHook)` | Record compaction |
| `AppendCustomEntry(customType, data)` | Extension data (not in LLM context) |
| `AppendCustomMessageEntry(customType, content, display, details)` | Extension message (in LLM context) |
| `AppendSessionInfo(name)` | Set session display name |
| `AppendLabelChange(targetId, label)` | Set or clear a label |

### Read methods

| Method | Description |
|--------|-------------|
| `GetLeafID()` | Current leaf entry ID |
| `GetLeafEntry()` | Current leaf entry |
| `GetEntry(id)` | Entry by ID |
| `GetBranch(fromId?)` | Path from root to entry (or current leaf) |
| `GetTree()` | Full tree as `[]*SessionTreeNode` |
| `GetChildren(parentId)` | Direct children of an entry |
| `GetLabel(id)` | Label for an entry |
| `GetEntries()` | All entries (excludes header) |
| `GetHeader()` | Session header |
| `GetSessionName()` | Display name from latest `session_info` |
| `GetCwd()` / `GetSessionDir()` / `GetSessionID()` / `GetSessionFile()` | Session metadata |
| `IsPersisted()` | Whether session writes to disk |
| `BuildSessionContext()` | Build LLM context from current tree position |

### Branching methods

| Method | Description |
|--------|-------------|
| `Branch(entryId)` | Move leaf to an earlier entry |
| `ResetLeaf()` | Reset leaf to before first entry |
| `BranchWithSummary(entryId, summary, details, fromHook)` | Branch + append summary of abandoned path |
| `CreateBranchedSession(leafId)` | Extract the branch path into a new `.jsonl` file |
| `NewSession(parentSession?)` | Start a fresh session |
| `SetSessionFile(path)` | Switch to a different session file |

### Static functions

| Function | Description |
|----------|-------------|
| `List(cwd, sessionDir?)` | List all sessions in a directory |
| `MigrateFromLinear(ctx, legacyDir, treeDir)` | Migrate legacy flat sessions to tree format |

## Persistence

### Lazy writes

The session file is **not created** until the first assistant message is appended. This avoids creating files for sessions that never get a response (e.g. abandoned requests).

### Append-only

New entries are appended to the end of the JSONL file. Existing entries are never modified or deleted. The only exceptions are:
- **Migration** (v1 -> v2 -> v3) rewrites the file once
- **`CreateBranchedSession`** writes a new file with only the entries on the selected branch

### Crash safety

Each append is followed by `fsync`. If the process crashes mid-write, the JSONL file may have a partial last line, which is silently skipped on the next load.

## Migration

### From legacy formats

`MigrateFromLinear()` handles automatic migration:

| Source format | Description |
|---------------|-------------|
| `.json` files | Legacy `session.SessionManager` snapshots |
| `.jsonl` + `.meta.json` | Legacy `memory.JSONLStore` append-only logs |

The migration:
1. Reads all messages from the legacy file
2. Creates a tree session with linear parentId chain
3. If a summary exists, adds it as a `CompactionEntry`
4. Renames the legacy file to `.tree-migrated`

Migration is **idempotent** -- already-migrated files are skipped.

### Version migrations

| Migration | Description |
|-----------|-------------|
| v1 -> v2 | Add `id`/`parentId` tree structure to entries |
| v2 -> v3 | Rename `hookMessage` role to `custom` |

Version migrations are applied automatically when opening a session file and trigger a file rewrite.

---

## Concurrency

`SessionManager` uses a `sync.RWMutex`:
- Write methods (`Append*`, `Branch`, etc.) acquire a write lock
- Read methods (`Get*`, `BuildSessionContext`) acquire a read lock

The `Adapter` uses a separate `sync.RWMutex` for the manager map, with double-checked locking on `getOrCreate`.

## Integration with picoclaw

The tree session manager is integrated via `initSessionStore()` in `pkg/agent/instance.go`:

```
initSessionStore(dir)
    |
    +-- MigrateFromLinear(legacyDir, treeDir)  -- auto-migrate legacy sessions
    |
    +-- tree.NewAdapter(treeDir)               -- create tree-based SessionStore
    |
    +-- (fallback) initLegacyStore(dir)        -- if migration fails
```

The `Adapter` implements `session.SessionStore`, so the agent loop uses it without changes via `AddMessage`, `GetHistory`, `SetSummary`, `TruncateHistory`, etc.

For advanced tree operations (branching, labels, tree traversal), callers can access the underlying `SessionManager` via `adapter.Manager(sessionKey)`.
